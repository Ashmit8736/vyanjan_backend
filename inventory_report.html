Petpooja Inventory Integration Report

1. Project Overview

This web application is a comprehensive restaurant inventory, recipe mapping, production execution, and billing management system. It integrates the frontend interface built in React + Electron (for desktop execution) and the backend REST API powered by Node.js (Express) and MariaDB.

The primary core data flow operates as follows:

Item Master → Vendor Master → Purchase Order (PO) → GRN/Stock Purchase → Stock In → Recipe Mapping → POS Billing with Auto-Consumption → Stock Adjustment / Wastage → Reports & Dashboard

2. Database Schema & Mapped Data

To support transactions, tracking, and logs, the database schema was modified or introduced with new tables:

A. sales_invoices (POS Billing Transactions)

Records high-level details of billing events.

Column Name Data Type Key / Reference Description
id INT Primary Key (AI) Unique Invoice ID
branch_id INT - Identifies the branch conducting the transaction
invoice_number VARCHAR Unique Unique business invoice reference number
total_amount DECIMAL(10,2) - Total billing amount before/after discounts
payment_status VARCHAR - Payment status: 'paid', 'unpaid', 'pending'
created_at TIMESTAMP - Invoice generation timestamp

B. sales_invoice_items (POS Invoice Items Sold)

Tracks quantity and pricing of specific items in each billing transaction.

Column Name Data Type Reference Description
id INT Primary Key (AI) Item Sold ID
invoice_id INT Foreign Key -> sales_invoices.id Associated parent invoice
item_id INT Foreign Key -> items.id The sold finished menu item
quantity DECIMAL(10,2) - Quantity sold
price DECIMAL(10,2) - Unit selling price

C. wastage_records (Raw Material Wastage)

Logs raw material wastage events, calculating immediate financial loss and deducting quantities from stock.

Column Name Data Type Reference Description
id INT Primary Key (AI) Wastage Log ID
branch_id INT - Identifies the branch of wastage
raw_material_id INT Foreign Key -> raw_materials.id Wasted ingredient reference
quantity DECIMAL(10,3) - Wasted quantity entered in purchase unit
loss_amount DECIMAL(10,2) - Computed loss value: quantity * purchase_price
reason VARCHAR - Reason: 'Expired', 'Spilled', 'Spoiled', etc.
created_at TIMESTAMP - Timestamp when wastage was registered

D. stock_adjustments (Inventory Variance Audits)

Keeps track of physical stock audit overrides and records variance checks with reasons.

Column Name Data Type Reference Description
id INT Primary Key (AI) Audit ID
branch_id INT - Branch identifier
raw_material_id INT Foreign Key -> raw_materials.id Modified ingredient
previous_quantity DECIMAL(10,3) - Stock level prior to override (in consume units)
new_quantity DECIMAL(10,3) - Stock level post override (in consume units)
variance DECIMAL(10,3) - Difference: new_quantity - previous_quantity
reason VARCHAR - Reason: 'Counting Error', 'Pilferage', 'Damage', etc.
comments TEXT - Custom notes written by auditor
created_at TIMESTAMP - Timestamp of stock override action

E. production (In-House Cooking Production Records)

Maintains history of recipe conversion execution runs.

Column Name Data Type Reference Description
id INT Primary Key (AI) Production Run ID
branch_id INT - Branch identifier
item_id INT Foreign Key -> items.id Prepared final menu item
recipe_id INT Foreign Key -> recipes.id Active recipe formulation bill of materials
produce_quantity DECIMAL(10,3) - Yield volume produced
produce_unit_id INT Foreign Key -> units.id Yield Unit matching the recipe
status VARCHAR - Defaulted to 'completed'
created_by INT - User ID of the employee executing

3. Backend API Directory

Below is the catalog of REST endpoints added or refactored to support inventory functions:

Method API Endpoint Request Payload (JSON Body) Database Operations & Business Logic
POST /api/invoices/create { total_amount, payment_status, items: [{ item_id, quantity, price }] } Inserts invoice transactions and executes nested BOM recipe material calculations to deduct matching raw materials from raw_material_stock in a SQL transaction.
GET /api/invoices/list None Retrieves billing transactions history for the branch.
POST /api/wastage/create [{ raw_material_id, quantity, reason }] Calculates cost of loss per item using conversion factors and averages, writes bulk rows to wastage_records, and subtracts ingredients from raw_material_stock.
GET /api/wastage/list None Queries wastage entries joined with unit symbols formatted in purchase units.
DELETE /api/wastage/:id None (ID in params) Deletes the wastage record, converts the saved purchase-unit quantity back to consume-units, and increments raw_material_stock to restore stock.
POST /api/stock/stockOverride { raw_material_id, new_quantity, reason, comments } Fetches existing consume stock, updates it to new_quantity, computes variance, and logs details in stock_adjustments.
GET /api/stock/stockSummaryReport Query params: startDate, endDate, unitType Runs complex aggregations to reconstruct historical Opening, Purchase, Excess, Consumed, Wasted, Adjusted, and Closing stock balances scaled to "Purchase" or "Consume" unit types.
POST /api/recipe/update { item_id, item_quantity, item_unit_id, materials: [...] } Sets previous active recipes for this item to is_active = 0 and inserts the updated active recipe BOM.
PUT /api/item/update/:id { name, category, selling_price, item_unit_id } Updates item details in the master database.
DELETE /api/item/delete/:id None Soft-deletes the selected item by setting `is_active = 0` in the database.
GET /api/item/logs/:id None Retrieves item creation timestamp, recipe mapping timestamp, and historical production runs.

4. Frontend Screen Mappings

How the React components connect and interact with these APIs:

  • Create Invoice Screen (CreateInvoice.jsx): Sends billing rows to POST /api/invoices/create upon checking out. Auto-deducts recipe materials on the backend in real-time.
  • Wastage Log Panel (Wastage.jsx): Feeds bulk item lists to POST /api/wastage/create using purchase units. Allows reversing/deleting entries using DELETE /api/wastage/:id.
  • Stock Adjustment Panel (AvailableStock.jsx): Provides dropdowns for variance reasons and custom comment fields. Submits overrides to POST /api/stock/stockOverride.
  • Production Execution Console (ProductionExecution.jsx): Side-by-side checked items selection. Checks stock levels dynamically on the client, and sends production orders to POST /api/production/create using the recipe's specific yield unit (not selling unit).
  • Item Creation & Actions (ItemCreation.jsx):
    • Clipboard button opens Audit Logs Dialog fetching logs via GET /api/item/logs/:id.
    • Pencil button opens Edit Dialog updating attributes via PUT /api/item/update/:id.
    • Cross button soft-deletes row via DELETE /api/item/delete/:id.
  • Stock Summary Dashboard (StockSummary.jsx): Pulls consolidated logs using GET /api/stock/stockSummaryReport. Calculates physical stock differences dynamically.

5. Development Environment Auto-Restart

To improve development speed, the backend environment was configured with nodemon. By running the start script:
npm start
It starts Node with nodemon watching file formats .js, .json. When any file edits are saved, the server auto-restarts, immediately deploying changes to port 5000.

पेटपूजा इन्वेंटरी एकीकरण रिपोर्ट

1. प्रोजेक्ट का अवलोकन (Project Overview)

यह वेब एप्लिकेशन एक व्यापक रेस्टोरेंट इन्वेंटरी, रेसिपी मैपिंग, प्रोडक्शन निष्पादन और बिलिंग प्रबंधन प्रणाली है। यह React + Electron (डेस्कटॉप निष्पादन के लिए) में निर्मित फ़्रंटएंड इंटरफ़ेस और Node.js (Express) और MariaDB द्वारा संचालित बैकएंड REST API को एकीकृत करता है।

मुख्य डेटा प्रवाह निम्नलिखित प्रकार से संचालित होता है:

आइटम मास्टर (Item) → वेंडर मास्टर (Vendor) → परचेज आर्डर (PO) → GRN/स्टॉक परचेज → स्टॉक इन → रेसिपी मैपिंग → POS बिलिंग (ऑटो-कंजम्पशन) → स्टॉक समायोजन/वेस्टेज → रिपोर्ट्स और डैशबोर्ड

2. डेटाबेस स्कीमा और मैप किया गया डेटा

लेनदेन, ट्रैकिंग और लॉग्स को रिकॉर्ड करने के लिए डेटाबेस स्कीमा में सुधार किए गए हैं और नई टेबल्स जोड़ी गई हैं:

A. sales_invoices (POS बिलिंग लेनदेन)

बिलिंग लेनदेन की बुनियादी जानकारी दर्ज करता है।

कॉलम का नाम डेटा टाइप कुंजी / संदर्भ विवरण
id INT प्राइमरी की (AI) विशिष्ट इनवॉइस आईडी
branch_id INT - शाखा (Branch) की पहचान
invoice_number VARCHAR विशिष्ट (Unique) इनवॉइस संदर्भ संख्या
total_amount DECIMAL(10,2) - कुल बिलिंग राशि
payment_status VARCHAR - भुगतान स्थिति: 'paid', 'unpaid', 'pending'
created_at TIMESTAMP - बिल बनने की तारीख और समय

B. sales_invoice_items (बेचे गए बिल आइटम्स)

प्रत्येक बिल लेनदेन में बेचे गए विशिष्ट आइटम्स की मात्रा और कीमत को ट्रैक करता है।

कॉलम का नाम डेटा टाइप संदर्भ (Reference) विवरण
id INT प्राइमरी की (AI) सोल्ड आइटम आईडी
invoice_id INT फॉरेन की -> sales_invoices.id संबद्ध मुख्य इनवॉइस
item_id INT फॉरेन की -> items.id बेचा गया फाइनल मेनू आइटम
quantity DECIMAL(10,2) - बेची गई मात्रा
price DECIMAL(10,2) - बेची गई दर (सेलिंग प्राइस)

C. wastage_records (कच्चे माल की बर्बादी/वेस्टेज)

कच्चे माल (रॉ मटेरियल) की बर्बादी को दर्ज करता है, वित्तीय नुकसान की गणना करता है और स्टॉक से मात्रा घटाता है।

कॉलम का नाम डेटा टाइप संदर्भ विवरण
id INT प्राइमरी की (AI) वेस्टेज लॉग आईडी
branch_id INT - वेस्टेज होने वाली शाखा की आईडी
raw_material_id INT फॉरेन की -> raw_materials.id बर्बाद हुए कच्चे माल की आईडी
quantity DECIMAL(10,3) - बर्बाद मात्रा (परचेज यूनिट जैसे kg/ltr में)
loss_amount DECIMAL(10,2) - नुकसान मूल्य: मात्रा * औसत खरीद मूल्य
reason VARCHAR - बर्बादी का कारण: 'Expired', 'Spilled', आदि
created_at TIMESTAMP - वेस्टेज दर्ज होने का समय

D. stock_adjustments (स्टॉक समायोजन/ऑडिट लॉग)

वास्तविक फिजिकल स्टॉक और डेटाबेस स्टॉक में आने वाले अंतर (Variance) का ऑडिट विवरण रखता है।

कॉलम का नाम डेटा टाइप संदर्भ विवरण
id INT प्राइमरी की (AI) समायोजन लॉग आईडी
branch_id INT - शाखा पहचानकर्ता
raw_material_id INT फॉरेन की -> raw_materials.id समायोजित कच्चा माल
previous_quantity DECIMAL(10,3) - समायोजन से पहले का स्टॉक (कंज्यूम यूनिट में)
new_quantity DECIMAL(10,3) - समायोजन के बाद का नया स्टॉक (कंज्यूम यूनिट में)
variance DECIMAL(10,3) - स्टॉक अंतर: new_quantity - previous_quantity
reason VARCHAR - कारण: 'Counting Error', 'Pilferage', आदि
comments TEXT - ऑडिटर द्वारा लिखी गई टिप्पणी
created_at TIMESTAMP - स्टॉक सुधार की तारीख और समय

E. production (आन्तरिक प्रोडक्शन रन)

रेसिपी मैपिंग के आधार पर तैयार किए गए इन-हाउस फूड प्रोडक्शन रन का रिकॉर्ड रखता है।

कॉलम का नाम डेटा टाइप संदर्भ विवरण
id INT प्राइमरी की (AI) प्रोडक्शन रन आईडी
branch_id INT - शाखा पहचानकर्ता
item_id INT फॉरेन की -> items.id तैयार किया गया खाद्य आइटम
recipe_id INT फॉरेन की -> recipes.id प्रयुक्त एक्टिव रेसिपी फॉर्मूलेशन
produce_quantity DECIMAL(10,3) - उत्पादित कुल मात्रा
produce_unit_id INT फॉरेन की -> units.id रेसिपी के अनुसार प्रयुक्त यील्ड यूनिट
status VARCHAR - डिफ़ॉल्ट रूप से 'completed'
created_by INT - ऑपरेटर की यूजर आईडी

3. बैकएंड एपीआई सूची (Backend API Directory)

इन्वेंटरी और बिलिंग कार्यों का समर्थन करने वाले बैकएंड एंडपॉइंट्स की सूची:

विधि (Method) एपीआई एंडपॉइंट रिक्वेस्ट बॉडी (JSON) बिजनेस लॉजिक और डेटाबेस ऑपरेशन्स
POST /api/invoices/create { total_amount, payment_status, items: [...] } इनवॉइस डेटा इंसर्ट करता है और रेसिपी के अनुसार कच्चे माल की मात्रा को ऑटोमैटिकली डिडक्ट करता है।
GET /api/invoices/list None शाखा के बिलिंग लेनदेन का इतिहास देता है।
POST /api/wastage/create [{ raw_material_id, quantity, reason }] परचेस यूनिट से स्टॉक घटाने के लिए कनवर्टर लगाकर डेटा वेस्टेज लॉग में बल्क इंसर्ट करता है।
GET /api/wastage/list None वेस्टेज प्रविष्टियों की सूची परचेस यूनिट प्रतीकों के साथ प्रस्तुत करता है।
DELETE /api/wastage/:id None (आईडी पाथ में) वेस्टेज रिकॉर्ड हटाता है और स्टॉक में पहले डिडक्ट की गई मात्रा को वापस जोड़ (Add Back) देता है।
POST /api/stock/stockOverride { raw_material_id, new_quantity, reason, comments } स्टॉक ओवरराइड करता है और अंतर को गणना कर stock_adjustments में लॉग करता.
GET /api/stock/stockSummaryReport कुएरी: startDate, endDate, unitType Opening, Purchase, Consumed, Wastage, Adjustment और Closing स्टॉक का पूर्ण दैनिक रिपोर्ट डेटा देता है।
POST /api/recipe/update { item_id, item_quantity, item_unit_id, materials: [...] } पुरानी एक्टिव रेसिपी को डी-एक्टिवेट करके नई सामग्री सूची इंसर्ट करता है।
PUT /api/item/update/:id { name, category, selling_price, item_unit_id } मास्टर डेटाबेस में आइटम की जानकारी अपडेट करता है।
DELETE /api/item/delete/:id None आइटम को सॉफ्ट-डिलीट (`is_active = 0`) करता है।
GET /api/item/logs/:id None आइटम क्रिएशन डेट, रेसिपी क्रिएशन डेट और पुराने प्रोडक्शन रन लॉग करता है।

4. फ़्रंटएंड स्क्रीन मैपिंग (Frontend Mappings)

रिएक्ट घटक इन APIs के साथ इस प्रकार जुड़ते हैं:

  • Create Invoice (CreateInvoice.jsx): बिल सेव होने पर POST /api/invoices/create को कॉल करता है, जिससे कच्चे माल की मात्रा बैकएंड पर तुरंत कम हो जाती है।
  • Wastage Screen (Wastage.jsx): वेस्टेज प्रविष्टि जोड़ने पर POST /api/wastage/create और रद्द करने पर DELETE /api/wastage/:id का उपयोग करता है।
  • Stock Adjustment (AvailableStock.jsx): स्टॉक अपडेट करते समय ओवरराइड डेटा POST /api/stock/stockOverride पर भेजता है।
  • Production Execution (ProductionExecution.jsx): रेसिपी की यील्ड यूनिट (जैसे `Pcs`) प्रदर्शित करता है और POST /api/production/create के जरिए प्रोडक्शन रन शुरू करता है।
  • Item Creation (ItemCreation.jsx):
    • पहला बटन (Clipboard): GET /api/item/logs/:id के जरिए ऑडिट लॉग दिखाता है।
    • दूसरा बटन (Pencil): PUT /api/item/update/:id के द्वारा आइटम एडिट करता है।
    • तीसरा बटन (Cross): DELETE /api/item/delete/:id के जरिए सॉफ्ट-डिलीट करता है।
  • Stock Summary Report (StockSummary.jsx): GET /api/stock/stockSummaryReport से दैनिक डेटा प्राप्त करता है और अंतर की गणना करता है।

5. विकास पर्यावरण ऑटो-रिस्टार्ट कॉन्फ़िगरेशन

विकास को आसान बनाने के लिए बैकएंड में nodemon कॉन्फ़िगर किया गया है। अब टर्मिनल पर केवल:
npm start
चलाने से nodemon सर्वर को फाइलों में होने वाले बदलावों के लिए ट्रैक करता है। जैसे ही आप कोई फाइल सेव करेंगे, सर्वर तुरंत ऑटो-रिस्टार्ट होकर पोर्ट `5000` पर अपडेट हो जाएगा।
